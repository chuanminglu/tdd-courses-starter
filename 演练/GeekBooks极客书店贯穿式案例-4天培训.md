## 📖 文档说明

### 文档目的

本文档为4天AI赋能研发运维培训提供一个**完整贯穿的实战案例**，通过GeekBooks极客书店项目，将10个培训模块串联起来，让学员在同一个业务场景中体验AI赋能的完整DevOps流程。

### 与演练手册的关系

- **本文档**: 提供项目背景、能力矩阵、10个模块演练提纲（提纲挈领）
- **演练手册**: 提供详细的提示词模板、操作步骤、参考答案（如《GeekBooks智能推荐功能_学员演练手册.md》）

**使用建议**: 讲师基于本文档设计课程逻辑，结合演练手册进行实战操作。

### 核心设计理念

**方案D: 五维能力矩阵**
- 将"AI赋能"拆解为5个能力维度，每个模块侧重不同维度
- 避免"提示词疲劳"，让学员建立立体化的AI能力

**方案E: 项目贯穿**
- 用GeekBooks项目贯穿4天课程，而非碎片化演练
- Sprint 1-2已完成 → Sprint 3智能推荐功能 → 全流程AI赋能实践

### 1.1 GeekBooks极客书店简介

**公司定位**: 面向程序员和技术爱好者的专业技术图书电商平台

**核心用户**:
- 👨‍💻 软件工程师（占比60%）
- 📊 数据分析师（占比20%）
- 🎓 计算机专业学生（占比15%）
- 🔬 技术爱好者（占比5%）

**业务规模**:
- 📚 图书品类：15,000+ SKU（覆盖编程、算法、AI、云计算等领域）
- 👥 注册用户：50万+
- 📦 月均订单：8万单
- 💰 年营收：5000万+

**技术架构**:
- **后端**: Spring Boot 3.x + Java 17
- **前端**: Vue 3 + TypeScript + Element Plus
- **数据库**: MySQL 8.0 + Redis 6.x
- **DevOps**: GitLab CI/CD + Docker + Kubernetes
- **监控**: Prometheus + Grafana

**团队组成**:
- 产品经理：1名（负责需求分析和产品规划）
- 架构师：1名（负责技术选型和架构设计）
- 后端开发：3名（2名高级 + 1名中级）
- 前端开发：3名（全部中级）
- 测试工程师：1名（熟悉自动化测试）
- DevOps工程师：1名（负责CI/CD和运维）

### 1.3 Sprint 3 智能推荐功能需求

#### 产品负责人的原始需求

> "我们发现很多用户在首页停留时间很短，直接去搜索框找书。
> 
> 我希望能做一个**智能推荐功能**，就是根据用户的浏览历史和购买记录，给他推荐合适的书籍。
> 
> 这样用户可以更容易发现新书，我们的GMV（交易额）也能提升。
> 
> 这个功能很重要，**必须在这个Sprint做完**！"

#### 业务目标

1. **核心目标**: 提升首页停留时长和点击率，优化用户发现新书的体验
2. **量化目标**: 
   - 首页停留时长提升至90秒+（提升100%）
   - 推荐点击率≥30%
   - GMV提升10-15%
3. **时间约束**: 2周Sprint内完成MVP版本

#### 当前痛点

- ❌ 需求描述模糊（"合适的书籍"是什么标准？）
- ❌ 缺少验收标准（如何判断推荐"成功"？）
- ❌ 没有技术约束说明（算法复杂度、性能要求）
- ❌ 工作量未知（可能超出Sprint容量）
- ❌ 缺少冷启动策略（新用户如何推荐？）

#### 真实约束条件

**团队能力约束**:
- ❌ 团队无算法工程师（禁止使用复杂ML模型）
- ✅ 团队熟悉Spring Boot、Vue 3、MySQL、Redis

**技术栈约束**:
- ❌ 禁止引入新中间件（如Elasticsearch、Kafka）
- ❌ 禁止修改数据库表结构（影响其他模块）
- ✅ 只能使用现有技术栈

**时间约束**:
- ⏰ Sprint周期：2周（10个工作日）
- ⏰ 团队速率：20-25 Story Points
- ⏰ 本Sprint已承诺修复2个P0 Bug（占用3 SP）

**数据约束**:
- ✅ 已有用户浏览埋点（最近30天，约10万条记录）
- ⚠️ 数据质量一般（20%缺少停留时长）
- ❌ 无数据仓库（数据分析需查询生产库）

### 3.1 第1天模块（模块一到模块四）

#### 模块一：AI+DevOps核心理念与技术栈

**演练任务**: 分析GeekBooks的数字化转型需求，设计AI工具链初步方案

**能力目标**（侧重场景识别）:
- ⭐⭐⭐⭐⭐ **场景识别**: 识别GeekBooks项目中哪些环节适合AI赋能
- ⭐⭐⭐ **工具链整合**: 设计六层AI工具链的初步方案
- ⭐⭐ **提示词工程**: 理解AI工具的基本使用方法

**演练步骤**:
1. **场景识别**（30分钟）:
   - 分析GeekBooks当前痛点（首页停留时长低、缺少个性化推荐）
   - 识别AI可赋能的场景（需求分析、架构设计、代码开发、测试、运维）
   - 评估ROI（哪些场景投入产出比最高？）

2. **工具链设计**（20分钟）:
   - Layer 1 - 模型层：选择合适的大模型（DeepSeek/ChatGPT/客户定制模型）
   - Layer 2 - 通用工具层：配置通用AI工具（元宝、CherryStudio）
   - Layer 3 - 研发工具层：集成VSCode/Cursor
   - Layer 4 - AI助手层：配置GitHub Copilot/Continue
   - Layer 5 - 建模工具层：配置Mermaid/PlantUML
   - Layer 6 - MCP服务层：选择MCP服务（Context7/Playwright等）

3. **团队讨论**（10分钟）:
   - 哪些AI工具适合我们团队？
   - 哪些场景AI价值最大？
   - 如何避免过度依赖AI？

**验收标准**:
- ✅ 识别至少5个AI可赋能的场景（需求/设计/开发/测试/运维）
- ✅ 设计六层AI工具链方案（每层至少选择1个工具）
- ✅ 评估每个场景的ROI（投入时间 vs 预期收益）

**交付物**:
- AI工具链配置清单（Excel/Markdown表格）
- 场景识别分析表（场景名称、AI价值、ROI评估）

#### 模块二：AI赋能的PLPT提示词工程

**演练任务**: 基于GeekBooks智能推荐功能，练习RTGO框架提示词编写

**能力目标**（侧重提示词工程）:
- ⭐⭐⭐⭐⭐ **提示词工程**: 掌握RTGO框架，能独立编写高质量提示词
- ⭐⭐⭐⭐ **工具链整合**: 熟练使用DeepSeek/ChatGPT等AI工具
- ⭐⭐⭐ **质量把控**: 使用三闸口框架减少AI幻觉

**演练步骤**:
1. **RTGO框架学习**（15分钟）:
   - R - Role: 如何定义AI的角色？
   - T - Task: 如何描述任务和输入材料？
   - G - Goal: 如何说明目标与意图？
   - O - Objective: 如何定义输出要求？

2. **提示词编写实战**（30分钟）:
   - 任务1：编写"需求澄清"提示词（将PO的原始需求转化为结构化需求）
   - 任务2：编写"用户故事生成"提示词（生成符合INVEST原则的Story）
   - 任务3：编写"技术方案设计"提示词（生成推荐算法选型方案）

3. **Few-shot学习练习**（10分钟）:
   - 提供1-2个优秀示例
   - 让AI按照示例格式生成新内容

4. **三闸口幻觉检测**（5分钟）:
   - 第一闸口：输入验证（输入材料是否完整？）
   - 第二闸口：输出验证（输出是否符合要求？）
   - 第三闸口：人工复核（是否有明显错误？）

**验收标准**:
- ✅ 编写至少3个符合RTGO框架的提示词
- ✅ 每个提示词AI输出质量达标（幻觉率<10%）
- ✅ 能使用Few-shot技巧优化提示词

**交付物**:
- 3个RTGO提示词（Markdown格式）
- AI输出结果（包含质量评估）

**能力目标**（侧重工具链整合）:

#### 模块十：运维阶段的AI赋能IT故障分析

**演练任务**: 模拟推荐服务故障，使用AI进行故障分析和RCA报告生成

**能力目标**（侧重质量把控）:
- ⭐⭐⭐⭐⭐ **质量把控**: 验证AI故障分析的准确性
- ⭐⭐⭐⭐ **工具链整合**: Prometheus + Grafana + AI
- ⭐⭐⭐⭐ **团队协作**: 运维团队应急响应

**演练步骤**:
1. **故障场景模拟**（10分钟）:
   - 场景：推荐服务响应时间突然上升至5秒
   - 现象：用户投诉首页加载慢、推荐点击率下降

2. **AI日志分析**（20分钟）:
   - 使用AI分析应用日志（识别异常请求）
   - 使用AI分析数据库慢查询日志
   - 使用AI分析Redis日志

3. **AI根因分析**（20分钟）:
   - 故障时间线重建（何时开始？影响范围？）
   - 5 Whys分析（为什么响应时间上升？）
   - 鱼骨图分析（人员/流程/技术/环境）

4. **AI生成RCA报告**（20分钟）:
   - 故障摘要
   - 故障时间线
   - 根因分析
   - 临时修复措施
   - 长期改进方案

**验收标准**:
- ✅ 根因分析准确（如：Redis缓存失效导致数据库查询压力过大）
- ✅ RCA报告完整（摘要/时间线/根因/修复/改进）
- ✅ 改进方案可执行（如：增加Redis缓存容量、优化SQL查询）

**交付物**:
- RCA报告（Root Cause Analysis Report）
- 改进方案（含实施计划和预期效果）
