# 需求澄清阶段 - 五维能力培养指南

## 📋 文档说明

本文档详细说明如何通过"需求澄清（AI辅助问答）"环节培养学员的五维能力：

- **提示词工程** ⭐⭐⭐
- **工具链整合** ⭐⭐⭐⭐
- **质量把控** ⭐⭐⭐⭐⭐
- **场景识别** ⭐⭐⭐
- **团队协作** ⭐⭐⭐

---

## 🎯 学习目标

通过需求澄清环节，学员应掌握：

1. ✅ 使用RTGO框架编写需求澄清提示词
2. ✅ 识别需求模糊点并生成结构化问题清单
3. ✅ 基于MoSCoW方法判定问题优先级
4. ✅ 整合AI工具（ChatGPT/Claude/Copilot）高效完成需求分析
5. ✅ 把控澄清结果质量（完整性、可追溯性、可验证性）

## 📊 五维能力培养详解

### 一、提示词工程能力 ⭐⭐⭐

#### 能力目标

学员能够独立编写高质量的需求澄清提示词，确保AI生成的问题清单符合业务场景和技术约束。

#### 培养步骤

**步骤1：理解RTGO框架（15分钟）**

- 📖 **学习材料**: 阅读 `.github/copilot-instructions.md` 中的RTGO框架说明
- 🎯 **练习目标**: 理解Role、Task、Goal、Objective四要素的作用
- ✅ **检查点**: 能用自己的话解释为什么需要Goal部分（而不是直接跳到输出要求）

**步骤2：分析示例提示词（20分钟）**

- 📖 **学习材料**: 阅读 `prompts/01-需求澄清提示词_GeekBooks智能推荐.md`
- 🎯 **练习目标**: 识别提示词中的4个核心要素（R/T/G/O）
- ✅ **检查点**: 完成以下表格

| RTGO要素 | 在示例提示词中的位置 | 核心内容摘要 | 为什么这样设计 |
|----------|---------------------|-------------|---------------|
| Role | 第X行 | 资深需求分析师，10年经验... | 建立AI专业可信度 |
| Task | 第X行 | 生成需求澄清问题清单 | 明确具体任务边界 |
| Goal | 第X行 | 2小时内完成需求澄清会议 | 说明业务价值和成功标准 |
| Objective | 第X行 | 6大类问题，不少于18条 | 定义输出结构和质量要求 |

**步骤3：识别提示词质量要点（15分钟）**

- 🎯 **练习任务**: 对照RTGO质量检查清单，检查示例提示词

```markdown
## RTGO框架质量检查清单

### R - Role (角色定义)
- [x] ✅ 角色具体（资深需求分析师，不是泛泛的"专家"）
- [x] ✅ 有年限/经验量化（10年互联网产品需求分析经验）
- [x] ✅ 擅长能力与Task对齐（需求澄清、优先级评估、技术可行性评估）
- [x] ✅ 建立专业权威性

### T - Task (任务描述)
- [x] ✅ 使用明确的动作动词（生成）
- [x] ✅ 说明任务对象（需求澄清问题清单）
- [x] ✅ 列出所有输入材料（PO需求、业务背景、团队约束、数据资源）
- [x] ✅ 提供必要的任务上下文（敏捷需求开发流程第一步）

### G - Goal (目标与意图)
- [x] ✅ 有核心目标（2小时内完成需求澄清会议）
- [x] ✅ 有具体目标（需求完整性、优先级明确、技术可行性、验收标准、风险识别）
- [x] ✅ 有业务价值（为PO/开发/测试/PM/用户带来的价值）
- [x] ✅ 有成功标准（≥12条问题，MUST≥6条，可直接用于会议）
- [x] ✅ Goal与Task对齐

### O - Objective (输出要求)
- [x] ✅ 有输出结构（6大部分：元信息、功能、非功能、约束、验收、风险）
- [x] ✅ 有质量要求（问题针对性、优先级合理性、可追溯性）
- [x] ✅ 有格式规范（Markdown表格、emoji、编号规则）
- [x] ✅ 有特别说明（提问技巧、会议时间分配、信息不足处理）
- [x] ✅ 有输出格式示例（表格模板）
```

**步骤4：动手修改提示词（30分钟）**

- 🎯 **练习任务**: 假设需求场景变更为"图书推荐改为课程推荐"，修改提示词
- ✅ **检查点**: 
  - Role部分是否需要调整？（如增加"在线教育产品经验"）
  - Task的输入材料是否需要调整？（如增加"课程分类体系"）
  - Goal的业务价值是否需要调整？（如"为学员"替换"为用户"）
  - Objective的问题分类是否需要调整？（如增加"学习路径"维度）

#### 能力检验

- ✅ **检验1**: 能独立编写一个新场景的需求澄清提示词（如"订单支付功能"），并通过RTGO质量检查清单
- ✅ **检验2**: 能识别提示词中的常见问题（如Goal缺失、输入材料不完整）并提出改进建议

### 二、工具链整合能力 ⭐⭐⭐⭐

#### 能力目标

学员能够熟练使用多种AI工具（GitHub Copilot、ChatGPT、Claude）完成需求澄清任务，并掌握工具间的协作流程。

#### 培养步骤

**步骤1：选择合适的AI工具（10分钟）**

- 📖 **学习材料**: 了解不同AI工具的特点

| AI工具 | 适用场景 | 优势 | 劣势 | 本任务推荐度 |
|--------|----------|------|------|-------------|
| GitHub Copilot Chat | 在IDE内快速生成文档 | 集成度高，上下文感知强 | 输出长度受限 | ⭐⭐⭐⭐ |
| ChatGPT-4 | 生成长文档、复杂分析 | 输出长度长，推理能力强 | 需要手动粘贴上下文 | ⭐⭐⭐⭐⭐ |
| Claude | 分析长文档（如会议纪要） | 支持100K+ token输入 | 响应速度较慢 | ⭐⭐⭐ |

- 🎯 **练习任务**: 基于本任务特点（需要生成18+条问题清单），选择最合适的工具
- ✅ **推荐答案**: **ChatGPT-4**（输出长度足够，推理能力强，适合生成结构化清单）

**步骤2：准备工具输入（15分钟）**

- 🎯 **练习任务**: 将以下材料整合到工具输入中

**输入清单**:
```markdown
1. 需求澄清提示词（prompts/01-需求澄清提示词_GeekBooks智能推荐.md）
2. 产品负责人原始需求（GeekBooks极客书店需求描述.md § 1.3）
3. 业务背景（GeekBooks极客书店需求描述.md § 1.1-1.2）
4. 埋点数据样例（data/user_browse_events.csv）
5. 埋点数据说明（data/埋点数据说明.json）
```

**整合方法**:
- ✅ **方法1**: 使用Markdown格式拼接所有材料
- ✅ **方法2**: 使用分隔符（如`---`）区分不同材料
- ✅ **方法3**: 优先提供提示词，再提供其他材料作为上下文

**步骤3：执行工具调用（20分钟）**

- 🎯 **练习任务**: 在ChatGPT中执行以下操作

**操作流程**:
```
1. 粘贴提示词（完整的RTGO内容）
2. 粘贴输入材料（PO需求、业务背景、团队约束、数据资源）
3. 发送请求
4. 检查输出是否符合要求
5. 如不符合，追加澄清指令（如"请增加2条关于数据质量的澄清问题"）
```

- ✅ **检查点**: 输出的问题清单是否包含：
  - 不少于18条问题
  - 按优先级分类（MUST/SHOULD/COULD）
  - 每个问题有提问理由和期望答案类型
  - 使用Markdown表格格式

**步骤4：工具间协作（15分钟）**

- 🎯 **练习任务**: 使用多工具协作提升质量

**协作流程**:
```
1. 使用ChatGPT生成初版问题清单（18条）
2. 将问题清单粘贴到GitHub Copilot，要求：
   - "请检查这些问题是否覆盖了冷启动策略"
   - "请检查是否有重复问题"
   - "请检查优先级判定是否合理"
3. 基于Copilot反馈，在ChatGPT中追加改进指令
4. 最终整合到需求文档中
```

**步骤5：结果导出与版本管理（10分钟）**

- 🎯 **练习任务**: 将AI生成的内容保存到版本库

**导出操作**:
```bash
# 1. 保存AI生成的问题清单
mkdir -p 演练/需求开发与管理/outputs
# 将ChatGPT输出复制到 outputs/需求澄清问题清单_v1.md

# 2. 提交到Git
git add outputs/需求澄清问题清单_v1.md
git commit -m "feat: 生成GeekBooks智能推荐需求澄清问题清单"

# 3. 记录生成过程
# 在 outputs/生成记录.md 中记录：
# - 使用的AI工具：ChatGPT-4
# - 使用的提示词版本：v1.0
# - 生成时间：2025-12-02
# - 追加指令（如有）：无
```

#### 能力检验

- ✅ **检验1**: 能在10分钟内选择合适的AI工具并准备好输入材料
- ✅ **检验2**: 能使用多工具协作（ChatGPT + Copilot）提升输出质量
- ✅ **检验3**: 能将AI生成内容规范化保存到版本库

### 三、质量把控能力 ⭐⭐⭐⭐⭐

#### 能力目标

学员能够识别AI生成的需求澄清问题清单中的质量问题，并进行有效的质量改进。

#### 培养步骤

**步骤1：理解质量标准（20分钟）**

- 📖 **学习材料**: 需求澄清问题清单的5大质量维度

| 质量维度 | 质量标准 | 常见问题 | 检查方法 |
|----------|----------|----------|----------|
| **完整性** | 覆盖功能、非功能、约束、验收、风险5大类 | 遗漏非功能需求澄清 | 对照5大类逐一检查 |
| **针对性** | 每个问题针对具体的需求模糊点 | 问题过于宽泛（如"如何实现？"） | 检查是否可直接回答 |
| **优先级合理性** | MUST问题不澄清就无法开发 | MUST问题过多（如15/18条） | 检查是否有默认假设 |
| **可追溯性** | 每个问题标注来源（PO需求/业务背景） | 缺少来源标注 | 检查是否有"（来源：...）" |
| **可操作性** | 问题可在2小时会议内回答完毕 | 问题过多（如30+条） | 计算会议时间分配 |

- 🎯 **练习任务**: 对照质量标准，检查AI生成的问题清单

**步骤2：识别常见质量问题（30分钟）**

- 🎯 **练习任务**: 分析以下劣质问题清单，识别质量问题

**劣质问题清单示例**:

| 问题ID | 优先级 | 问题类型 | 问题 | 提问理由 | 期望答案类型 |
|--------|--------|----------|------|----------|--------------|
| Q1 | ✅MUST | 功能需求 | 如何实现推荐功能？ | 需要了解实现方式 | 开放式 |
| Q2 | ✅MUST | 功能需求 | 推荐算法要达到什么效果？ | 需要明确目标 | 开放式 |
| Q3 | ✅MUST | 性能需求 | 性能要求是什么？ | 需要了解性能指标 | 开放式 |

**质量问题分析**:

- ❌ **问题Q1**: 过于宽泛，无法直接回答（应拆分为"推荐算法"+"推荐位置"+"推荐数量"）
- ❌ **问题Q2**: 缺少具体维度（应明确是"准确率"还是"点击率"还是"GMV"）
- ❌ **问题Q3**: 缺少具体指标（应明确"响应时间P95<X秒"）
- ❌ **通用问题**: 缺少来源标注、缺少备选答案、优先级判定可能有误

**改进后的问题清单**:

| 问题ID | 优先级 | 问题类型 | 问题 | 提问理由 | 期望答案类型 | 备选答案 |
|--------|--------|----------|------|----------|--------------|----------|
| Q1 | ✅MUST | 功能需求 | 推荐算法采用什么策略？ | PO需求"根据浏览历史和购买记录推荐"未说明算法类型（来源：PO需求第2段） | 封闭式 | A.协同过滤 B.内容推荐 C.热度排序 D.混合策略 |
| Q2 | ✅MUST | 功能需求 | 推荐位放在首页哪个位置？ | PO需求未说明推荐展示位置（来源：PO需求"首页停留时间短"） | 封闭式 | A.顶部Banner B.中部推荐位 C.底部 |
| Q3 | ✅MUST | 非功能需求 | 首页推荐加载时间上限是多少？（P95） | 技术债务"首页加载P95约2.5秒"，需明确推荐模块的性能要求（来源：业务背景§1.2） | 量化数据 | 建议<500ms |

**步骤3：执行质量改进（40分钟）**

- 🎯 **练习任务**: 对AI生成的问题清单执行以下质量改进

**改进清单**:

```markdown
## 质量改进检查表

### 1. 完整性检查
- [ ] 是否覆盖功能需求？（推荐算法、推荐位置、推荐数量、冷启动）
- [ ] 是否覆盖非功能需求？（性能、可用性、安全、监控）
- [ ] 是否覆盖技术约束？（数据来源、技术选型、接口设计）
- [ ] 是否覆盖验收标准？（业务指标、转化指标、技术指标）
- [ ] 是否覆盖风险识别？（时间风险、技术风险、数据风险）

### 2. 针对性检查
- [ ] 每个问题是否针对具体的需求模糊点？（非泛泛而谈）
- [ ] 每个问题是否可直接回答？（非"如何做到最好"）
- [ ] 每个问题是否有明确的期望答案类型？（开放式/封闭式/量化）

### 3. 优先级检查
- [ ] MUST问题是否不澄清就无法开发？（如推荐算法）
- [ ] SHOULD问题是否有默认假设？（如推荐数量默认10本）
- [ ] COULD问题是否可后续迭代？（如推荐解释功能）
- [ ] MUST问题占比是否合理？（建议40-50%，即8/18条）

### 4. 可追溯性检查
- [ ] 每个问题是否标注来源？（PO需求/业务背景/团队约束/数据资源）
- [ ] 来源标注是否具体到段落？（如"来源：PO需求第2段"）

### 5. 可操作性检查
- [ ] 问题总数是否合理？（建议15-20条，控制在2小时会议内）
- [ ] 每个问题预计讨论时间是否明确？（MUST:5-10分钟，SHOULD:3-5分钟）
```

**改进操作**:

1. **补充遗漏问题**: 如果某个质量维度未覆盖，补充2-3条问题
2. **拆分宽泛问题**: 如果问题过于宽泛，拆分为2-3个具体问题
3. **调整优先级**: 如果优先级判定不合理，重新评估并调整
4. **添加来源标注**: 为每个问题添加`（来源：...）`标注
5. **添加备选答案**: 为封闭式问题添加A/B/C/D选项

**步骤4：同行评审（20分钟）**

- 🎯 **练习任务**: 2人一组，互相评审问题清单

**评审表格**:

| 评审维度 | 评分（1-5分） | 问题点 | 改进建议 |
|----------|--------------|--------|----------|
| 完整性 | 4 | 缺少数据隐私相关问题 | 补充Q19：用户是否可关闭推荐功能？ |
| 针对性 | 5 | 无 | - |
| 优先级合理性 | 3 | MUST问题过多（12/18条） | 将Q5、Q8降为SHOULD |
| 可追溯性 | 4 | Q3、Q7缺少来源标注 | 补充来源标注 |
| 可操作性 | 5 | 无 | - |
| **总分** | **21/25** | - | - |

#### 能力检验

- ✅ **检验1**: 能在15分钟内完成质量改进检查表（识别至少5个质量问题）
- ✅ **检验2**: 能基于质量问题提出具体改进措施（非泛泛而谈）
- ✅ **检验3**: 能在同行评审中给出有价值的反馈（评分+问题点+改进建议）

### 四、场景识别能力 ⭐⭐⭐

#### 能力目标

学员能够识别需求澄清任务中的关键场景，并针对不同场景调整提示词和问题策略。

#### 培养步骤

**步骤1：识别需求澄清的典型场景（15分钟）**

- 📖 **学习材料**: 需求澄清的5种典型场景

| 场景类型 | 特征 | 澄清重点 | 提示词调整 |
|----------|------|----------|-----------|
| **新功能开发** | PO有初步想法，但细节模糊 | 功能边界、验收标准 | 增加"功能范围"澄清问题 |
| **性能优化** | 现有功能性能不满足需求 | 性能基线、目标值、测试方法 | 增加"当前性能数据"输入材料 |
| **技术债务修复** | 技术问题影响业务 | 影响范围、优先级、修复成本 | 增加"技术债务清单"输入材料 |
| **合规性需求** | 需满足法规/标准 | 合规标准、验证方法、审计要求 | 增加"合规标准文档"输入材料 |
| **用户体验改进** | 用户反馈体验问题 | 用户痛点、改进目标、A/B测试 | 增加"用户反馈数据"输入材料 |

- 🎯 **练习任务**: 识别"GeekBooks智能推荐"属于哪种场景
- ✅ **答案**: **新功能开发** + **用户体验改进**（组合场景）

**步骤2：识别场景中的关键约束（20分钟）**

- 🎯 **练习任务**: 从需求描述中提取关键约束

**约束识别表**:

| 约束类型 | 具体约束 | 来源 | 影响 | 应对策略 |
|----------|----------|------|------|----------|
| **团队能力约束** | 无算法工程师 | 业务背景§1.3 | 禁止复杂ML模型 | 采用基于规则的推荐算法 |
| **技术栈约束** | 禁止新增中间件 | 业务背景§1.3 | 无法使用Elasticsearch | 推荐结果缓存在Redis |
| **数据库约束** | 禁止改表结构 | 业务背景§1.3 | 无法新增推荐表 | 复用现有埋点表 |
| **时间约束** | 2周Sprint | 业务背景§1.3 | 需MVP版本 | 分阶段交付（一期基础推荐） |
| **数据质量约束** | 20%埋点缺失 | 数据资源 | 推荐准确率受影响 | 数据清洗+降级策略 |

- ✅ **检查点**: 识别出至少5个关键约束

**步骤3：场景化调整澄清策略（25分钟）**

- 🎯 **练习任务**: 基于约束调整问题清单

**调整策略**:

**原始问题**（未考虑约束）:
```
Q1: 推荐算法采用什么机器学习模型？
```

**调整后问题**（考虑"无算法工程师"约束）:
```
Q1: 推荐算法采用什么策略？
   备选答案：
   A. 基于用户浏览历史的协同过滤（ItemCF）
   B. 基于图书分类的内容推荐
   C. 基于浏览量的热度排序
   D. 混合策略（协同过滤+热度排序）
   
   约束说明：团队无算法工程师，禁止使用复杂ML模型（如深度学习）
```

**练习**: 调整以下问题

**问题Q5**（未考虑数据约束）:
```
Q5: 推荐算法需要哪些数据？
```

**调整后**（考虑"20%埋点缺失"约束）:
```
Q5: 推荐算法需要哪些数据？20%埋点数据缺少停留时长，如何处理？
   备选答案：
   A. 过滤掉缺失数据（数据量减少20%）
   B. 用平均停留时长填充（可能引入偏差）
   C. 仅使用浏览记录，不使用停留时长（降低推荐精度）
   
   约束说明：已有埋点数据质量一般，20%记录缺少停留时长字段
```

**步骤1：准备需求澄清会议（20分钟）**

**会议准备清单**:

```markdown
## 需求澄清会议准备清单

### 1. 会议组织

- [ ] 预约会议室（2小时，支持投影/白板）

### 2. 会议材料

- [ ] 准备白板笔（用于记录讨论结果）
- [ ] 准备便利贴（用于投票/标注）
- [ ] 准备录音设备（用于会后整理）

### 3. 会议议程

### 4. 角色分工
- [ ] 主持人：架构师/BA（控制会议节奏）
- [ ] 记录人：测试工程师/BA（记录澄清结果）
- [ ] 决策人：产品负责人（最终决策）
- [ ] 技术顾问：后端Lead/前端Lead（提供技术建议）
```

**步骤2：模拟需求澄清会议（40分钟）**

- 🎯 **练习任务**: 3-4人一组，模拟需求澄清会议

**角色分配**:
- **产品负责人**：提出需求，回答业务问题
- **架构师/BA**：主持会议，提问澄清问题
- **开发Lead**：提出技术约束，评估可行性
- **测试工程师**：记录澄清结果，提出验收标准

**会议流程**（按问题清单逐条讨论）:

```
1. 主持人读问题（如Q1：推荐算法采用什么策略？）
2. 产品负责人回答（如"我们希望用协同过滤，但不确定团队能否实现"）
3. 开发Lead评估可行性（如"ItemCF可以实现，但需要2-3天开发+测试"）
4. 讨论备选方案（如"一期先用热度排序，二期再优化为ItemCF"）
5. 记录人记录决策（如"决策：一期采用热度排序，二期迭代为ItemCF"）
6. 继续下一个问题
```

**会议记录模板**:

| 问题ID | 问题 | PO回答 | 技术评估 | 最终决策 | 备注 |
|--------|------|--------|----------|----------|------|
| Q1 | 推荐算法采用什么策略？ | 希望用协同过滤 | ItemCF可实现，需2-3天 | 一期：热度排序<br>二期：ItemCF | 分阶段交付 |
| Q2 | 推荐位放在首页哪个位置？ | 中部推荐位 | 需前端调整布局 | 中部推荐位（Banner下方） | 需UI设计配合 |
| ... | ... | ... | ... | ... | ... |
