## 模块3：需求开发与管理（详细演练）

目标：生成BRD/PRD、用户画像、用户旅程、可交付的原型说明与需求追踪矩阵。

时长建议：90-120 分钟

能力侧重（五维）

- 提示词工程：⭐⭐⭐
- 工具链整合：⭐⭐⭐⭐
- 质量把控：⭐⭐⭐⭐⭐
- 场景识别：⭐⭐⭐
- 团队协作：⭐⭐⭐

步骤与练习：

1. 输入准备

   - 提供PO原始需求、业务背景、已有埋点数据样例（CSV/JSON）
   - 提供团队约束（无算法工程师、禁止新增中间件等）
2. 需求澄清（AI辅助问答）

   - 使用RTGO模板生成澄清问题清单（不少于12条）
   - 优先级判定（MUST/SHOULD/COULD）
3. BRD 编写

   - 生成context diagram
   - 生成BRD结构：背景、问题陈述、目标与KPI、用户画像、核心流程、成功标准
   - 明确量化验收指标（首页停留时间、推荐点击率、GMV增幅等）
4. 业务流程梳理（15-20 分）

   - 基于BRD文档
   - 识别主流程关键节点（至少7个）：用户访问首页 → 加载历史数据 → 调用推荐服务 → 展示推荐结果 → 用户点击 → 记录埋点 → 结束
   - 识别异常场景（至少3个）：推荐服务超时、新用户冷启动、缓存失效
   - 使用Mermaid绘制流程图（包含决策点、异常处理、降级策略）
   - 输出：
     - Mermaid流程图代码
     - 流程说明（200字）
     - 异常场景说明（每个异常包含：触发条件、处理策略、降级方案、用户影响）
   - 提示词参考：`prompts/03-业务流程梳理提示词_通用版.md`
5. 需求拆解演练（20 分）

   5.1 用例法（10 分）**【3阶段工作流】**

   - 能力侧重：场景识别⭐⭐⭐、提示词工程⭐⭐⭐
   - 基于BRD文档，使用3阶段工作流完成用例法演练：

   **阶段1：用例识别与编写**（3分钟）

   - 识别2-3个核心用例
   - 编写用例文档（用例描述、参与者、前置/后置条件、主场景、替代场景、业务规则、验收标准）
   - 提示词参考：`prompts/04-1-用例识别与编写提示词_通用版.md`

   **阶段2：场景拆解**（3分钟）

   - 基于用例文档，拆解出详细场景列表（主场景、替代场景、异常场景、边界场景）
   - 每个用例至少拆解5个场景
   - 提示词参考：`prompts/04-2-场景拆解提示词_通用版.md`

   **阶段3：测试用例生成与追溯矩阵**（4分钟）

   - 基于场景列表，生成测试用例
   - 建立需求追溯矩阵（Use Case → Scenario → Test Case）
   - 计算覆盖率（P0场景100%、P1场景≥80%、P2场景≥50%）
   - 提示词参考：`prompts/04-3-测试用例生成与追溯矩阵提示词_通用版.md`
   - 最终输出：

     - 用例文档（2-3个核心用例）
     - 场景列表（每个用例5-8个场景）
     - 测试用例列表（P0/P1场景全覆盖）
     - 需求追溯矩阵（Use Case → Scenario → Test Case）
     - 覆盖率统计报告

   5.2 用户故事法（10 分）**【重点演练】【4阶段工作流】**

   - 能力侧重：提示词工程⭐⭐⭐⭐、工具链整合⭐⭐⭐⭐⭐、质量把控⭐⭐⭐⭐⭐、场景识别⭐⭐⭐⭐、团队协作⭐⭐⭐⭐
   - 基于BRD，使用4阶段工作流完成Epic分析和用户故事拆解：

   **阶段1：Epic分析与策略制定**（2分钟）

   - 识别用户角色、制定拆解策略、评估依赖关系和风险
   - 输出：Epic分析报告（Epic概述、拆解策略、用户角色分析、依赖关系、风险识别）
   - 提示词参考：`prompts/05-1-Epic分析与策略制定提示词_通用版.md`

   **阶段2：用户故事生成**（3分钟）

   - 基于Epic分析结果，生成10-15个用户故事（标准格式：作为...我希望...以便...）
   - 输出：用户故事列表（按功能分组，包含故事描述、功能要点、业务规则）
   - 提示词参考：`prompts/05-2-用户故事生成提示词_通用版.md`

   **阶段3：INVEST评估与优先级排序**（3分钟）

   - 对用户故事进行INVEST原则评估（独立性、可协商、有价值、可估算、小粒度、可测试）
   - 使用MoSCoW方法进行优先级分类（Must/Should/Could/Won't Have）
   - 输出：INVEST评估表 + MoSCoW优先级分类 + Sprint交付计划
   - 提示词参考：`prompts/05-3-INVEST评估与优先级排序提示词_通用版.md`

   **阶段4：验收标准编写**（2分钟）

   - 为Must Have的高优先级故事编写详细验收标准（Given-When-Then格式）
   - 输出：验收标准文档（3-5个故事，包含验收标准、边界条件、性能要求、UI/UX要求）
   - 提示词参考：`prompts/05-4-验收标准编写提示词_通用版.md`
   - 最终输出：

     - Epic分析报告
     - 用户故事列表（10-15个，符合INVEST原则）
     - INVEST评估表 + MoSCoW优先级分类 + Sprint交付计划
     - 验收标准文档（3-5个高优先级故事）

   - 五维能力培养：参考 `05-用户故事拆解五维能力培养指南.md`
6. 原型生成练习（15-20分）**【3阶段工作流】**

   - 能力侧重：提示词工程⭐⭐⭐、工具链整合⭐⭐⭐⭐、质量把控⭐⭐⭐⭐
   - 方法学：基于《用户体验的要素》五层模型（战略层、范围层、结构层、框架层、表现层）进行原型设计
   - 基于用户故事列表，使用3阶段工作流完成原型生成练习：

   **阶段1：页面元素分析与提取**（5分钟）

   - 从用户故事列表中选择1-3个典型Must Have故事
   - 基于《用户体验的要素》战略层、范围层、结构层，提取页面结构和UI组件
   - 输出：
     - 核心页面清单（页面ID、页面名称、页面类型、功能描述、导航关系）
     - UI组件清单（组件ID、组件名称、组件类型、所属页面、交互说明）
     - 数据字段定义（字段ID、字段名称、数据类型、验证规则、示例值）
     - 数据契约（页面间数据传递 + API接口定义）
   - 提示词参考：`prompts/06-1-页面元素分析与提取提示词_通用版.md`

   **阶段2：交互设计与流程定义**（5分钟）

   - 基于页面元素分析报告，设计页面间的交互流程和跳转逻辑
   - 基于《用户体验的要素》框架层，定义交互行为和反馈设计
   - 输出：
     - 页面跳转流程图（PlantUML活动图代码）
     - 关键交互点说明（按钮点击效果、页面响应、数据传递、异常处理）
     - 交互反馈设计（成功/失败/加载/空状态的文案和样式）
     - 交互动画定义（页面切换、组件交互动画）
     - 异常场景处理（网络异常、数据无效、权限不足等场景的处理策略）
   - 提示词参考：`prompts/06-2-交互设计与流程定义提示词_通用版.md`

   **阶段3：HTML原型页面生成**（5-10分钟）

   - 基于页面元素和交互流程，生成可交互的HTML/CSS/JavaScript原型页面
   - 基于《用户体验的要素》表现层，实现现代化视觉设计
   - 输出：
     - 3-5个独立的HTML文件（P001-{页面名称}.html、P002-{页面名称}.html...）
     - 每个文件包含完整的HTML结构、内联CSS样式、内联JavaScript代码
     - 实现页面跳转、表单验证、Toast提示、Modal弹窗、加载状态等核心交互
     - 使用Ant Design/Element UI风格的现代化设计
   - 提示词参考：`prompts/06-3-HTML原型页面生成提示词_通用版.md`

   **最终输出**:
   - 输出1：Human-readable Prototype Spec（页面清单、UI组件清单、数据字段定义、交互流程图、交互说明、反馈设计）
   - 输出2：可交互的HTML/CSS/JavaScript原型页面（3-5个HTML文件）

   **验收标准**:
   - ✅ 原型覆盖3-5个核心页面（列表页、详情页、表单页等）
   - ✅ HTML文件可以直接在浏览器打开，无需服务器
   - ✅ 页面间可以通过链接跳转
   - ✅ 表单验证逻辑完整（必填、格式、长度验证）
   - ✅ Toast提示和Modal弹窗正常工作
   - ✅ 交互流程图可以正常渲染（PlantUML）
   - ✅ 页面设计风格统一（Ant Design/Element UI风格）

交付物：

- 澄清问题清单（Markdown）
- BRD 文档（Markdown）
- 业务流程图（Mermaid代码 + 流程说明 + 异常场景说明）
- 用例文档（2-3个核心用例 + 需求追溯矩阵）
- 用户故事列表（10-15个，含Epic分析、INVEST评估、MoSCoW优先级、验收标准）
- 原型说明（结构化JSON + 人类可读规范）

质量检查点：

- BRD/PRD 中每个目标都有量化指标与验收方法
- 业务流程图包含至少7个关键节点、至少3个异常场景
- 异常场景有明确的降级策略（不能是"人工处理"等模糊表达）
- 用例文档包含主场景和至少2个替代场景，需求追溯矩阵完整
- 用户故事符合INVEST原则（评分≥80分）
- 用户故事格式规范（作为...我希望...以便...）
- MoSCoW优先级分类合理（Must Have≥6个故事）
- 验收标准使用Given-When-Then格式，具体可测试
- 原型说明包含关键交互与数据字段
